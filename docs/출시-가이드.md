# 케이스 1 — 실제 App Store 심사 제출

## 1-A. Xcode에서 최종 확인 (5분)

1. Xcode로 `TimeLock.xcodeproj` 열기
2. 프로젝트 → TimeLock 타깃 → **Signing & Capabilities** 탭:
   - "Automatically manage signing" 체크 + 팀 선택 확인
   - **Sign in with Apple**과 **Time Sensitive Notifications**가 capability 목록에 보이는지 확인. 안 보이면 `+ Capability` 버튼으로 각각 추가 (entitlements 파일엔 이미 있어서 추가하면 바로 붙음)
   - 빨간 서명 오류가 있으면: Xcode → Settings → Accounts에서 Apple ID 로그인 → "Download Manual Profiles"
3. 시뮬레이터 아무거나 골라 **⌘R 실행** — 빌드가 통과하고 온보딩이 뜨면 OK

## 1-B. App Store Connect — 앱 등록 (10분)

> https://appstoreconnect.apple.com → 나의 앱 → `+` → 신규 앱

|---|---|
| 플랫폼 | iOS |
| 이름 | **앵그리모티** |
| 기본 언어 | 한국어 |
| 번들 ID | `com.selfer.angrymoti` (드롭다운에 없으면 아래 "번들 ID 먼저 등록" 박스 참고) |
| SKU | `angrymoti-ios` (내부 식별용, 사용자에게 안 보임) |
| 사용자 액세스 권한 | 전체 액세스 |

> 5. Continue → Register → App Store Connect 신규 앱 창을 새로고침하면 드롭다운에 나타남 (1~2분 걸릴 수 있음)

## 1-F. Firebase 콘솔 최종 세팅 (10분)

1. Authentication → 템플릿 → 보낸사람 이름: "앵그리모티 (AngryMoti)"
2. Firestore → 규칙 → 아래로 교체 후 게시:
   ```
   rules_version = '2';
   service cloud.firestore {
     match /databases/{database}/documents {
       match /users/{uid}/{document=**} {
         allow read, write: if request.auth != null && request.auth.uid == uid;
       }
       // 그룹 챌린지 — 로그인 사용자 전체 접근 (v1 신뢰 기반)
       match /groups/{roomID} {
         allow read, create, update, delete: if request.auth != null;
         match /members/{memberID} {
           allow read, write: if request.auth != null;
         }
       }
     }
   }
   ```
3. **(보안, 꼭)** Google Cloud 콘솔 → API 및 서비스 → 사용자 인증 정보 → iOS API 키 → "iOS 앱 제한"에 번들 ID `com.selfer.angrymoti` 등록. (저장소가 공개라 plist의 API 키가 노출돼 있음 — 번들 제한을 걸면 악용 불가)
4. **심사용 데모 계정 생성**: 앱에서 이메일 가입 → 인증 메일 클릭까지 완료한 계정 1개. (예: `review@yourdomain.com` / 비밀번호 기록)

# 케이스 2 — TestFlight 내부 테스트 (팀원 기기 배포)

> 내부 테스트는 **Apple 심사 없이** 업로드 즉시 배포 가능. 조건: 테스터가 App Store Connect 계정에 멤버로 등록돼 있어야 함 (최대 100명). 스크린샷·설명·구독 심사 전부 불필요.

## 2-A. 팀원을 App Store Connect에 초대 (5분)

1. App Store Connect → **사용자 및 액세스** → `+`
2. 팀원 이름 + Apple ID 이메일 입력, 역할은 **"고객 지원"** 이면 충분 (개발자 권한 불필요)
3. 팀원이 초대 메일 수락

## 2-B. 빌드 업로드 (케이스 1-G와 동일)

1. Any iOS Device (arm64) → Product → Archive → Distribute App → App Store Connect → Upload
2. 케이스 1을 이미 진행했다면 **같은 빌드를 그대로 재사용** — 다시 올릴 필요 없음
3. 빌드 처리 완료 메일이 오면 다음 단계로

> 참고: 첫 업로드 시 "수출 규정" 질문은 안 뜸 (`ITSAppUsesNonExemptEncryption=false` 반영돼 있음)

## 2-C. 내부 테스터 그룹 구성 (5분)

1. App Store Connect → 앱 → **TestFlight** 탭
2. 좌측 "내부 테스트" → `+` → 그룹 이름 예: `팀 테스트`
3. 그룹에 빌드 선택 + 2-A에서 초대한 팀원 체크
4. 팀원에게 자동으로 TestFlight 초대 메일 발송됨

## 2-D. 팀원이 할 일 (각자 2분)

1. App Store에서 **TestFlight** 앱 설치
2. 초대 메일의 "View in TestFlight" 탭 → 수락 → 설치
3. 이후 새 빌드를 올릴 때마다 TestFlight 앱에 자동으로 업데이트 뜸

## 2-E. 테스트할 때 알아둘 것

- **구독 결제는 샌드박스**: TestFlight에서 멤버십을 구독해도 실제 과금 없음. 샌드박스 구독은 갱신 주기가 압축됨 (월 구독 → 5분마다 갱신, 몇 번 후 자동 만료) — 갱신·만료 테스트에 오히려 유용
- **알람 테스트**: 실기기에서 알림 권한 허용 필수. 무음 스위치를 켜도 알람음이 나는지, 잠금 화면에서 알림이 뜨는지 확인
- **크래시 리포트**: TestFlight 앱에서 스크린샷+피드백을 바로 보낼 수 있음 (App Store Connect → TestFlight → 피드백에서 수집)
- 새 빌드를 올릴 땐 Xcode에서 **빌드 번호만 +1** (General 탭 → Build: 1 → 2). 버전(1.0.0)은 그대로 둬도 됨

## 권장 순서

```
[지금] 유료 앱 계약 + 은행 등록 신청 (반영 대기 시간이 제일 김)
  ↓
[오늘] 케이스 2로 TestFlight 내부 배포 → 팀원 실기기 테스트 (1~2일)
  ↓
[테스트 중 발견된 문제 수정] → 빌드 번호 +1 재업로드
  ↓
[문제 없으면] 케이스 1의 나머지 (스크린샷·메타데이터·구독 상품·심사 노트) → 제출
```

같은 빌드가 TestFlight와 심사 제출에 공용이라, 내부 테스트에서 통과한 빌드를 그대로 심사에 넣는 게 가장 안전해.
